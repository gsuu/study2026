# 5월 스터디 리뷰 기준

이 문서는 상태 중심 사고 훈련 스터디를 운영할 때 사용할 리뷰 기준입니다.

핵심은 코드 스타일을 평가하는 것이 아니라,  
상태 정의, 파생값 구분, 변경 지점 통제, 구조 설명 가능성을 기준으로 판단하는 것입니다.

## 리뷰 원칙

- 취향보다 구조를 봅니다.
- 동작 여부보다 상태 모델을 먼저 봅니다.
- "예쁘게 짰다"보다 "흐름이 읽힌다"를 더 중요하게 봅니다.
- 문제를 지적할 때는 반드시 이유를 같이 설명합니다.
- 가능하면 "무엇을 먼저 고쳐야 하는가"까지 제안합니다.

## 리뷰 순서

리뷰는 아래 순서로 진행합니다.

1. 진짜 state가 무엇인지 확인
2. derived value를 state에 저장했는지 확인
3. 상태 변경 지점이 어디에 모였는지 확인
4. `render()` 밖에서 UI를 직접 만지는지 확인
5. 책임이 섞여 있는지 확인
6. 설명이 가능한 구조인지 확인

## 핵심 리뷰 질문

### 상태

- 이 값은 진짜 state인가?
- 줄일 수 있는 state가 있는가?
- 같은 의미의 값을 여러 개 들고 있지 않은가?

### 파생값

- 계산 가능한 값을 state에 저장하고 있지 않은가?
- 총 개수, 총 가격, 선택된 객체 같은 값은 파생값으로 처리할 수 없는가?

### 흐름

- 상태 변경은 어디서 일어나는가?
- 변경 지점이 여러 군데 흩어져 있지 않은가?
- 클릭 핸들러마다 UI를 직접 맞추고 있지 않은가?

### render

- DOM 반영이 `render()`에 모여 있는가?
- `classList`, `style`, `textContent`를 상태 저장소처럼 쓰고 있지 않은가?
- source of truth가 DOM이 아니라 state인가?

### 책임

- 한 함수가 여러 책임을 동시에 갖고 있지 않은가?
- 상태 변경, 계산, DOM 반영이 한 곳에 섞여 있지 않은가?

### 설명 가능성

- 왜 이 상태가 필요한지 설명할 수 있는가?
- 왜 이 값은 state가 아니고 파생값인지 설명할 수 있는가?
- 이 구조를 팀원이 처음 읽어도 흐름을 따라갈 수 있는가?

---

## 좋은 코드의 기준

### 좋은 코드

- 상태가 적고 명확합니다.
- 파생값을 저장하지 않습니다.
- 상태 변경 지점이 통제됩니다.
- `render()`가 UI 반영 책임을 가집니다.
- 읽을 때 흐름이 보입니다.
- 왜 이렇게 설계했는지 설명할 수 있습니다.

### 나쁜 코드

- 상태가 많고 의미가 겹칩니다.
- 계산 가능한 값을 저장합니다.
- DOM을 직접 수정하면서 상태처럼 사용합니다.
- 이벤트마다 제각각 UI를 바꿉니다.
- 변경 지점이 여러 군데 흩어져 있습니다.
- 흐름을 설명하기 어렵습니다.

---

## 감점 포인트

아래 항목은 리뷰에서 바로 잡아야 합니다.

- `isOpen`, `isClosed` 같이 같은 의미의 상태를 여러 개 둠
- 총 가격, 결과 개수, 필터 결과 목록을 state에 저장함
- 이벤트 핸들러에서 `classList`, `style.display`, `textContent`를 직접 계속 수정함
- DOM class를 읽어서 현재 상태를 판단함
- 한 함수 안에서 상태 변경, 계산, DOM 반영을 전부 처리함
- 상태 변경 지점이 파일 여러 곳에 흩어져 있음

## 리뷰 코멘트 예시

### 상태 관련

- "이 값은 state로 들고 갈 필요가 없어 보여요. 계산으로 얻을 수 있습니다."
- "여기 `isOpen` 하나면 충분한데 의미가 같은 값이 두 개 있습니다."
- "지금 source of truth가 DOM class로 가 있어서 상태가 숨어 있습니다."

### 파생값 관련

- "총 가격은 저장하지 말고 `cartItems`에서 계산하는 게 맞습니다."
- "결과 개수는 파생값이라 state 불일치 위험이 있습니다."

### 흐름 관련

- "열고 닫는 로직이 여러 이벤트 핸들러에 흩어져 있어서 추적이 어렵습니다."
- "변경 지점을 `setState()` 또는 action 함수로 모아야 합니다."

### render 관련

- "여기서는 상태만 바꾸고 DOM 반영은 `render()`에서 처리하는 편이 더 낫습니다."
- "지금 `classList`가 상태처럼 쓰이고 있어서 상태 모델이 약해집니다."

### 책임 관련

- "이 함수는 상태 변경, 가격 계산, UI 반영을 동시에 하고 있습니다."
- "책임을 나누면 읽기도 쉬워지고 버그도 줄어듭니다."

## 회차별 리뷰 초점

### 1주차

- 진짜 state를 찾았는가
- derived value를 구분했는가
- boolean 남발을 줄였는가

### 2주차

- DOM 직접 제어를 끊었는가
- source of truth를 state로 옮겼는가
- 상태 변경 지점을 모았는가

### 3주차

- 왜 이 코드가 별로인지 구조 기준으로 설명할 수 있는가
- 책임이 섞인 함수를 분해할 수 있는가
- 최소 수정안이 구조적으로 맞는가

### 4주차

- 설계와 구현이 일치하는가
- state와 derived value가 잘 분리됐는가
- 리팩토링이 기능 추가가 아니라 구조 개선으로 이어졌는가

## 리뷰 진행 방식

### DAY 1

- 코드 리뷰보다 설계 리뷰를 먼저 합니다.
- 상태 정의와 데이터 흐름 설명을 중심으로 봅니다.
- 이 시점에는 구현 디테일보다는 사고 구조를 봅니다.

### DAY 2

- 구현 후 구조 리뷰를 합니다.
- 최소 정답이 맞는지 먼저 보고, 그 다음 실무형 확장 포인트를 봅니다.
- 리팩토링이 필요한 지점은 하나씩 좁혀서 제안합니다.

## 이 문서의 역할

- 리더는 이 문서로 리뷰 기준을 통일합니다.
- 팀원은 이 문서로 좋은 코드와 나쁜 코드를 판별하는 기준을 익힙니다.
- 커리큘럼 문서와 실습 교안을 실제 운영에 연결하는 기준 문서로 사용합니다.
