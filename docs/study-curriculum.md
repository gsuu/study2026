# 5월 스터디 커리큘럼

## Astro + 상태 중심 사고 훈련

### 한 줄 목표

DOM을 직접 조작하는 개발자에서  
상태를 먼저 설계하는 개발자로 전환합니다.

### 이 스터디가 바꾸려는 것

이 스터디는 JavaScript 문법을 많이 가르치는 과정이 아닙니다.  
핵심은 아래 3가지를 팀의 기본 습관으로 만드는 것입니다.

1. 코딩 전에 상태부터 정의합니다.
2. DOM 직접 제어 코드를 상태 기반으로 리팩토링합니다.
3. 코드 품질을 상태 기준으로 설명합니다.

### 사용 기술

- Astro
- 바닐라 JavaScript
- island는 4주차에서 정말 필요할 때만 최소 사용

### 운영 원칙

- 설계 없이 구현하지 않습니다.
- 상태 정의 없이 UI를 만들지 않습니다.
- 이벤트 핸들러에서 DOM을 직접 맞추는 습관을 줄입니다.
- `state -> setState -> render` 흐름으로 사고합니다.
- 계산 가능한 값은 state로 저장하지 않습니다.
- 구현 개수보다 상태 판단력과 설명 능력을 더 중요하게 봅니다.

### 스터디 운영 방식

- 주 2회 진행
- `DAY 1`: 설계와 사고훈련 중심
- `DAY 2`: 구현과 리팩토링 중심

### DAY 1 규칙

- 코드 작성 금지
- 상태 정의 먼저 작성
- 파생값 구분 필수
- 데이터 흐름 설명 필수
- 설계 발표 후 피드백 진행

### DAY 2 규칙

- DAY 1 설계를 바탕으로만 구현
- 구현 후 구조 리뷰 진행
- 반드시 한 번은 리팩토링 수행
- 기능 추가보다 구조 개선 우선

### 필수 제출 형식

모든 과제와 훈련은 아래 순서로 제출합니다.

1. 상태 정의
2. 상태 변경 이벤트
3. 파생값 정의
4. UI 반영 방식
5. 리팩토링 또는 구현 결과
6. 왜 이렇게 설계했는지 설명

## 1주차: 상태 사고

### 목표

화면을 먼저 보지 않고,  
무슨 상태가 있어야 하는지 먼저 생각하는 습관을 만듭니다.

### DAY 1

- 사고훈련: 요구사항을 state로 번역하기
- 사고훈련: state와 derived value 구분하기
- 사고훈련: 중복 상태 줄이기

### DAY 2

- 구현 과제: 탭 UI
- 핵심 질문
  - 진짜 state는 무엇인가?
  - boolean 여러 개 대신 하나의 상태로 줄일 수 있는가?
  - 버튼 스타일과 패널 표시가 같은 상태를 보고 있는가?

### 1주차 결과

- `이건 상태다 / 이건 파생값이다`를 구분할 수 있어야 합니다.

---

## 2주차: 상태 리팩토링

### 목표

지금 팀이 자주 쓰는  
제이쿼리식 바닐라 JS와 DOM 직접 제어 코드를 상태 기반으로 바꾸는 훈련을 합니다.

### DAY 1

- 리팩토링 훈련: 탭 잘못된 코드 고치기
- 리팩토링 훈련: 아코디언 잘못된 코드 고치기
- 리팩토링 훈련: 모달 잘못된 코드 고치기

### DAY 2

- 리팩토링 훈련: 드롭다운 잘못된 코드 고치기
- 리팩토링 훈련: 필터 리스트 잘못된 코드 고치기
- 핵심 질문
  - source of truth가 DOM인가 state인가?
  - 변경 지점이 흩어져 있는가?
  - `render()` 밖에서 UI를 직접 만지고 있는가?

### 2주차 결과

- DOM을 맞추는 코드와 상태를 설계하는 코드의 차이를 설명할 수 있어야 합니다.

## 3주차: 코드 품질 판단

### 목표

좋은 코드와 나쁜 코드를  
상태 기준으로 판별하는 언어를 팀 안에 만듭니다.

### DAY 1

- 사고훈련: 잘못된 코드에서 문제 지점 찾기
- 사고훈련: 중복 상태, 파생값 저장, 이벤트 분산 찾기
- 사고훈련: 책임이 섞인 함수 분해하기

### DAY 2

- 리뷰 훈련: 문제 코드 비교 리뷰
- 리뷰 훈련: 최소 수정안 제시하기
- 핵심 질문
  - 상태가 과한가?
  - 파생값을 저장하고 있는가?
  - 흐름이 한 번에 읽히는가?
  - 상태 변경 지점이 통제되는가?

### 3주차 결과

- 왜 이 코드가 별로인지를 취향이 아니라 구조 기준으로 말할 수 있어야 합니다.

## 4주차: 통합 적용

### 목표

지금까지 훈련한 사고를  
설계 -> 구현 -> 리팩토링으로 끝까지 연결합니다.

### DAY 1

- 통합 설계 과제: 미니 커머스 UI
- 범위
  - 상품 리스트
  - 카테고리 필터
  - 정렬
  - 장바구니
  - 총 가격
  - 상세 모달

### DAY 2

- 구현 과제: 미니 커머스 UI
- 리뷰 및 리팩토링
- 핵심 질문
  - 무엇이 진짜 state인가?
  - 총 가격은 왜 state가 아닌가?
  - 모달과 장바구니 상태는 왜 분리되는가?
  - 상태 변경 지점은 어디에 모였는가?

### 4주차 결과

- 기능을 만든 것보다, 설계와 흐름을 설명할 수 있어야 성공입니다.

---

## 강제 규칙

- 설계 없는 구현 금지
- state 정의 없는 구현 금지
- 설명 못 하는 state 금지
- 파생값 저장 습관 감점
- 이벤트 핸들러에서 DOM 직접 맞추기 감점
- 같은 의미의 상태 여러 개 두기 감점

## 이 문서의 역할

- 리더는 이 문서로 운영 기준을 잡습니다.
- 팀원은 이 문서로 학습 방향과 제출 기준을 이해합니다.
- 이후 문서인 `실습 교안`, `리뷰 기준 문서`는 이 커리큘럼을 기준으로 확장합니다.
